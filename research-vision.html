Logic,in Dialogue

A Research Vision · Originals & Taylor's Versions

Four questions about how humans and formal systems exchange answers, and what changes when a language model joins the loop

Overture

Formal systems produce facts about the world. A type checker proves an invariant. A model checker rules out a state. A rule engine returns an outcome. A theorem prover certifies a derivation. Each fact is small, checkable, and grounded in logic. Each one stands on its own.

The harder problem is what happens next. A fact has to reach a person who will act on it. The person speaks natural language. The system speaks logic. The two languages do not align. A developer reading an analyzer report, a tax preparer asking a rule engine about an edge case, a mathematician inspecting a proof assistant: each one is doing the same work, translating between an answer the system computed and a question the person actually had in mind.

My research has been about that translation, organized around four questions. Why did the system do what it did. How do you present the answer to a human. What does the human contribute back. Was the exchange productive. The first three are design questions. The fourth is an empirical one. I have worked on these in different roles since 2006, in academia, at SRI, and at AWS, with software as the running case.

Since 2024 the assumption underneath the loop has changed. The human in the loop was always the limiting reagent. Crowds were slow. Annotations were expensive. Reports went unread. Large language models lift that constraint, and in doing so they reopen every research question that was settled around the old constraint. It is worth going back through the work and asking which results still hold, which methods become newly tractable, and which protocols need to be designed from scratch.

I am now in my neurosymbolic era. The catalog below pairs published work with a Taylor's Version that asks how it changes when a language model joins the loop.

Language models change what plays the human role. The fluent participant is now cheap, fast, and available at every step of the loop. It can read the artifact, propose the missing piece, and explain a finding in plain English. It can also be confidently wrong in ways formal systems cannot. So the four questions reopen, and they reopen wherever a formal system meets a human audience. The interpolant explaining a program failure to a coding agent. The conditional answer a rule engine has to give when a question leaves most of the input blank. The proof step a theorem prover has to justify in prose for the mathematician at the keyboard. The same loop, different artifacts. The interesting research question is which parts of the symbolic side bear weight when the rest is rebuilt.

The Catalog

Four sides, one question each. The published work appears on top. Below it, a Taylor's Version asks how the same question reads when a language model is in the room. The catalog leans on software because that is where most of the published work lives. The questions do not.

Side A · Why?

Why Did the System Do That?

Distilling a failing run, in logic, into the smallest fact that explains it.

A1 · 2012

Error Invariants

E. Ermis, M. Schäf, T. Wies — FM

A sequence of Craig interpolants along a failing trace produces the smallest formula at each program point that explains why the trace fails. The result is a logical fingerprint of the bug, one slice at a time.

A2 · 2013

Flow-sensitive Fault Localization

J. Christ, E. Ermis, M. Schäf, T. Wies — VMCAI

Fault localization that respects control flow. The explanation tracks how information moves through branches and joins, so the localized cause is faithful to the program's structure rather than a flat slice.

A3 · 2016

Classifying Bugs with Interpolants

A. Podelski, M. Schäf, T. Wies — TAP

The shape of an interpolant is itself a fingerprint of the bug class. Quantifiers, variables, and arithmetic operators carry information about what kind of error produced the trace. Off-by-one looks like off-by-one in logic.

Side A · Taylor's Version

Why, Re-recorded

The symbolic backbone stays in place. The participant on the generative side changes. Error Invariants becomes a hypothesis loop. The language model proposes candidate explanations in natural language. The interpolation engine accepts only those backed by a valid interpolant. Flow-sensitive Fault Localization becomes a comparison. Does an LLM reading of a failing trace respect control flow the way a flow-sensitive symbolic localizer does, or does it produce plausible flat narratives that miss the join points? Classifying Bugs with Interpolants becomes a head-to-head with LLM embeddings of the same buggy programs. The interpolant signature is small, sound, and interpretable. The embedding is large, fluent, and opaque. The question is which bug families each one captures and which neither does.

From the Vault When the symbolic and generative sides disagree on what to highlight, which signal teaches the developer faster, and which bug families does neither representation catch?

Side B · Show?

How Do You Say It to a Human?

A correct fact is not yet a useful one. Presentation is its own research problem.

B1 · 2015

Bixie: Finding and Understanding Inconsistent Code

T. McCarthy, P. Rümmer, M. Schäf — ICSE Demo

A tool that finds inconsistent code, meaning paths the program will execute that contradict its own assumptions. The contradiction is rendered in a form a developer can read, not just one a verifier can prove.

B2 · 2015

VERMEER

D. Schwartz-Narbonne, C. Oh, M. Schäf, T. Wies — ICSE Demo

An interactive tool that traces and explains C-program failures via concolic abstraction and slicing. The audience is a human reading a counter-example and deciding what the program had been trying to do.

B3 · 2016

Multistaging to Understand

H. Sanchez, E. J. Whitehead, M. Schäf — ICPC

Distilling Java code examples into staged simplifications. Each stage drops details a reader can defer, leaving a kernel that captures the example's essence. Comprehension as a controlled descent through abstraction levels.

Side B · Taylor's Version

Show, Re-recorded

Each of these tools chose what not to show. Bixie hid the parts of the proof a developer did not need. VERMEER abstracted the trace to the slice that mattered. Multistaging dropped detail in stages until the kernel surfaced. With a language model in the loop, the abstraction can be rendered in prose. Bixie becomes a tool whose contradictions are explained in natural language. The verifier releases only the explanations it can re-derive from the inconsistency. VERMEER reverses its audience. The explanation now flows to an LLM coding agent. The new dependent variable is how aggressively the trace can be abstracted before the agent loses the bug. Multistaging to Understand becomes a question about who chooses the stages. An LLM-chosen simplification may still be faithful, or it may drop the detail that carried the meaning.

From the Vault Are humans and language models sensitive to the same abstractions, or do the same explanations land for one audience and fall flat for the other?

Side C · Listen?

What Does the Other Voice Add?

When the verifier gets stuck, someone else gets to guess. The protocol turns guesses into something checkable.

C1 · 2014

Chekofv: Crowd-Sourced Formal Verification

H. Logas, F. Kirchner, J. Murray, M. Schäf, J. Whitehead — FWFM

A game in which crowds propose candidate loop invariants and a verifier filters them. The protocol works because humans can guess what verifiers cannot synthesize.

C2 · 2016

Crowd-Sourcing Program Preconditions

D. Fava, D. Shapiro, J. Osborn, M. Schäf, J. Whitehead — ICSE

Players label test inputs. Preconditions are inferred from their consensus. A precondition, in this view, is the boundary a crowd draws around valid.

C3 · 2017

Abduction by Non-Experts

N. Bjørner, D. Jovanović, T. Lepoint, P. Rümmer, M. Schäf — LPAR

Empirical study showing that laypeople can perform abductive reasoning, given an interface that translates the symbolic problem into something familiar.

C4 · 2020

Continuous Compliance & Verifying Object Construction

M. Kellogg et al. — ASE & ICSE

Two type-system-based checkers. One enforces compliance properties on every commit. One tracks object initialization. Both are sound. Both pay the same tax: an annotation only the developer can write. The "other voice" here is quieter, but it is the same voice.

Side C · Taylor's Version

Listen, Re-recorded

The crowd protocol consists of a generative human and a symbolic referee. The re-recording puts a language model on the generative side. Chekofv becomes an LLM ensemble proposing invariants while the verifier rejects the unsound. The new question is whether the ensemble preserves the diversity of guess that made the crowd useful, or collapses onto a narrow modal hypothesis. Crowd-Sourcing Preconditions swaps a hundred labelers for one model under many personas. The question is whether the inferred boundary still bends around the surprising edge cases. Abduction by Non-Experts becomes a replication with an LLM as the non-expert under study. The interesting cases are abductive failures the model shares with human novices and abductive failures unique to the model. The type-system pair is the same protocol at a different tempo. The model proposes the qualifiers no one wants to write. The type checker rejects the unsound ones. The rejections become a curriculum. The annotations write themselves. The checker stays the unmoved gatekeeper.

From the Vault When the verifier rejects, does the model iterate productively, or does it argue with the verifier in confident, well-formed sentences?

Side D · Measure?

Was Any of This Productive?

The unfashionable side. Production studies that ask whether the loop converges on anything useful.

D1 · 2023

Long-term Rule Quality via True Negatives

L. Luo, R. Mukherjee, O. Tripp, M. Schäf, Q. Zhou, D. Sanchez — ICSE-SEIP

Methodology for monitoring static-analysis rule quality over time using true-negative measurements. In industrial settings, precision and recall on a fixed benchmark are a comforting lie.

D2 · 2024

Developer–Analyzer Interactions in Code Reviews

B. Cirisci, L. Luo, M. N. Mansur, D. Sanchez, M. Schäf, O. Tripp, M. B. Zafar, Q. Zhou — ASE

An empirical study of how developers respond to analyzer findings on pull requests. What they accept, what they ignore, what they argue with.

Side D · Taylor's Version

Measure, Re-recorded

The empirical strand becomes more interesting, not less, when a language model is in the loop. Long-term Rule Quality via True Negatives applies the same monitoring methodology to LLM-based checks. The drift signature changes shape. Models go stale overnight in ways static rules cannot. The true-negative measurement becomes the only honest way to notice. Developer-Analyzer Interactions grows a third party. An LLM mediator translates findings, summarizes them, and proposes fixes. The observational methodology stays the same. Three voices now appear on every pull request. The empirical question is whether the mediator is a net win for developer trust or a new register of confident wrongness.

From the Vault How do empirical methods built around static, sound analyzers adapt to a world where the analyzer itself can change between Tuesday and Wednesday?

Coda

The Bridge: Tracks From the Current Era

One released, two in the studio. The neurosymbolic era, in progress.

Released · ICSE-SEIP 2026

Delta Debugging for LLM-integrated Systems

Zhu, Mansur, Schäf, Chen, Lepoint, Visser

Delta debugging is a 1999 algorithm. It minimizes a failing input until the smallest one that still fails is found. The paper applies it to systems whose failing component is a language model. The four questions still apply. Why did it fail: the minimal witness. How to show it: a printable counter-example. What does the system add back: each minimization step. Was it productive: yes, measurably, on real LLM-integrated systems. Old machinery, new artifact, same conversation.

In the Studio · Vault Track 1

A neurosymbolic primitive for question answering over rule engines

The artifact is not a program. It is a rule engine, of the kind that underlies contracts, tax software, and eligibility checks. The engine answers point queries cleanly. It answers human questions badly, because the human leaves most of the input form blank and the engine cannot say what depends on the missing parts. The paper introduces a neurosymbolic primitive that walks the rule-trigger graph, constructs a small covering set of inputs, and renders a conditional answer of the form "yes if X, no if Y" that re-executes through the engine for validation. The four questions show up together. Side A distills the relevant dependency. Side C accepts the model's hypotheses through a verifier. Side B presents the result in a form a human can read. Side D measures whether the result was correct. The same loop, in a different discipline.

In the Studio · Vault Track 2

Learned eligibility conditions, validated on production billing

Pricing-term oracles produce thousands of false discrepancies on real billing data. The reason is that the eligibility conditions live in source code and side-channel docs, not in the terms themselves. An LLM coding agent learns the missing conditions from the data, validates each one against production billing with zero false positives, and traces it back to its documentation source. If no source exists, the agent flags it as a documentation gap. The methodology is Side D in full bloom. The analyzer is now the agent. The true-negative monitoring carries forward. The productivity question is answered nightly.
